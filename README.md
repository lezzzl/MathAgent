# MathAgent

MathAgent for summer MLA.

## Запуск сервера модели

### Команда запуска

Дефолты рассчитаны на выделенный сервер с A100-80GB:

```bash
vllm serve Qwen/Qwen2.5-7B-Instruct \
  --max-model-len 32768 \
  --max-num-seqs 32 \
  --gpu-memory-utilization 0.90 \
  --enable-auto-tool-choice \
  --tool-call-parser hermes
```

`--max-num-seqs` — жёсткий предел на стороне сервера: даже если клиент пришлёт
больше запросов, лишние встанут в очередь вместо раздувания KV-кэша. Держите его
не ниже `workers × sample-workers`.

### Расчёт памяти под KV-кэш

vLLM резервирует память **на старте** по `--gpu-memory-utilization`, поэтому
потребление VRAM в рантайме постоянно: лишние запросы встают в очередь
планировщика, а не раздувают память. Клиент не может вызвать OOM на GPU — если
сервер поднялся, он держит нагрузку.

Расчёт для Qwen2.5-7B (GQA, ~56 KiB KV на токен) на A100-80GB при
`--gpu-memory-utilization 0.90`: 72 ГБ пула минус 15.2 ГБ весов = **56.8 ГБ под
KV**, то есть ёмкость ~1.06 млн токенов.

| Контекст | KV на последовательность | Помещается параллельно |
|---|---|---|
| 8192 | 0.44 ГБ | 130 |
| 16384 | 0.88 ГБ | 65 |
| 32768 | 1.75 ГБ | 32 |

Это худший случай, когда каждая последовательность занимает весь контекст.
На типичных сэмплах (~3000 токенов) помещается порядка 355 запросов, так что
`workers × sample-workers = 16` использует единицы процентов пула.

На машине послабее считайте так же: `workers × sample-workers × контекст ×
56 KiB` не должно превышать пул за вычетом весов. Раннер печатает эту оценку
на старте (`[budget]`).

Без `--enable-auto-tool-choice --tool-call-parser hermes` модель не сможет
вызывать `python_exec`: она напишет вызов текстом, а агент это заметит и
напечатает `[TOOL PARSING BROKEN]`.

Контекст меньше ~6000 токенов заметно бьёт по качеству: сэмплы обрезаются, не
дойдя до `\boxed`, и теряют голос при голосовании (следите за
`n_truncated_samples` в результатах).

Перед стартом раннер сам опрашивает `/v1/models`: проверяет доступность
сервера, наличие запрошенной модели и предупреждает, если `max_model_len`
меньше 16384. При недоступном сервере прогон завершается с кодом 2, не создавая
файл результатов. Если сервер умирает посреди прогона, после двух подряд задач
без единого ответа раннер останавливается и подсказывает команду `--resume`.

## Режимы решения

| Режим | Что делает | Когда использовать |
|---|---|---|
| `--role sc` (по умолчанию) | N независимых решений, мажоритарное голосование по `\boxed` | Основной режим |
| `--role solver` | Пошаговый LangGraph: generate → evaluate → branch/recovery → verify | Эксперименты с пошаговым поиском |

### Self-consistency

```bash
python scripts/benchmarks/run_aime26.py \
  --model Qwen/Qwen2.5-7B-Instruct \
  --base-url http://127.0.0.1:8000/v1 \
  --role sc --n-samples 8 --sample-workers 4 --workers 4 --max-tokens 3000
```

Полезные поля в `metadata.agent_metrics`:

* `votes` — распределение голосов;
* `agreement` — доля сэмплов за победивший ответ. Хороший прокси уверенности:
  задачи с низким согласием почти всегда решены неверно, и по ним имеет смысл
  доливать сэмплы;
* `n_truncated_samples` — сколько сэмплов упёрлось в `--max-tokens` и не дошло
  до `\boxed`. Если это число не ноль, поднимайте лимит: такие сэмплы просто
  теряют голос.

Общее число одновременных запросов к серверу — `workers × sample-workers`.

### Параллельный прогон и возобновление

Задачи решаются в `--workers` потоков. Прогон можно продолжить после обрыва:

```bash
python scripts/benchmarks/run_aime26.py --resume results/aime26/agent_20260719T182437Z.jsonl
```

Уже решённые `task_id` пропускаются; записи с `metadata.error` перерешиваются.

## Инструмент `python_exec`

Многострочный Python с предзагруженным sympy (`solve`, `Eq`, `Rational`,
`factorint`, `isprime`, `divisors`, `binomial`, `Matrix`), numpy и itertools.
Имена sympy перекрывают `math`, поэтому `sqrt(8)` остаётся точным `2*sqrt(2)`.

Исполняется в отдельном долгоживущем процессе с таймаутом 10 с; на таймауте
процесс убивается и поднимается заново. Каждый вызов получает свежую копию
пространства имён, переменные между вызовами не протекают.

### Ресурсы песочницы

Процессы-песочницы живут в общем пуле и переиспользуются между задачами и
сэмплами. Их число ограничено сверху и **не зависит** от `workers ×
sample-workers`:

| Переменная | По умолчанию | Смысл |
|---|---|---|
| `SANDBOX_WORKERS` | 4 | Сколько процессов-песочниц держать (~200 МБ каждый) |
| `SANDBOX_MEMORY_MB` | 2048 | Потолок памяти на один вызов |

Таймаут ограничивает время, но не память: `list(range(10**9))` успевает съесть
десятки гигабайт за отведённые 10 секунд. Поэтому родитель следит за RSS
воркера и убивает его при превышении лимита, возвращая модели осмысленную
ошибку. Слежение требует `psutil`; без него остаётся только защита по времени.

Это **не** песочница безопасности: sympy и numpy дают достаточно, чтобы обойти
ограничение `__builtins__`. Изоляция здесь защищает от зависаний и случайных
ошибок модели, а не от злонамеренного кода.
